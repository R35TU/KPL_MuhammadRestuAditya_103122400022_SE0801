## Design Pattern yang Digunakan

Design pattern yang digunakan pada proyek ini adalah **Repository Pattern**.

Repository Pattern merupakan pola desain yang digunakan untuk memisahkan proses akses data dari logika bisnis aplikasi. Dengan pola ini, proses pengambilan, penyimpanan, pembaruan, dan penghapusan data dilakukan melalui sebuah repository sehingga bagian lain dari aplikasi tidak perlu mengetahui bagaimana data tersebut disimpan atau diambil.

## Potongan Kode

### base_repository.dart

```dart
abstract class BaseRepository<T> {
  Future<List<T>> getAll();
  Future<T?> getById(dynamic id);
  Future<T?> add(T item);
  Future<bool> updateItem(dynamic id, T item);
  Future<bool> deleteItem(dynamic id);
}
```

### project_repository.dart

```dart
class ProjectRepository implements BaseRepository<ProjectModel> {
  final String _table = 'projects';

  @override
  Future<List<ProjectModel>> getAll() async {
    final response = await supabase.from(_table).select();
    return response.map((json) => ProjectModel.fromJson(json)).toList();
  }

  @override
  Future<ProjectModel?> getById(dynamic id) async {
    final response = await supabase.from(_table).select().eq('id', id).maybeSingle();
    if (response == null) return null;
    return ProjectModel.fromJson(response);
  }

  @override
  Future<ProjectModel?> add(ProjectModel item) async {
    final response = await supabase.from(_table).insert(item.toJson()).select().maybeSingle();
    if (response == null) return null;
    return ProjectModel.fromJson(response);
  }
}
```

## Deskripsi

Pada proyek ini terdapat sebuah kelas abstrak `BaseRepository<T>` yang berisi deklarasi operasi-operasi dasar untuk melakukan pengelolaan data (CRUD), yaitu mengambil seluruh data, mengambil data berdasarkan ID, menambahkan data, memperbarui data, dan menghapus data.

Selanjutnya, kelas `ProjectRepository` mengimplementasikan seluruh fungsi tersebut untuk mengakses data proyek yang tersimpan pada database Supabase. Dengan demikian, bagian lain dari aplikasi cukup menggunakan `ProjectRepository` tanpa perlu mengetahui detail proses komunikasi dengan database.

---

## Penjelasan Design Pattern

Repository Pattern digunakan untuk memisahkan **lapisan akses data (data access layer)** dari **lapisan logika aplikasi (business logic)**.

Pada implementasi ini, `BaseRepository<T>` berperan sebagai kontrak (interface) yang mendefinisikan operasi CRUD yang harus dimiliki oleh setiap repository.

Kemudian `ProjectRepository` mengimplementasikan kontrak tersebut dan menangani seluruh proses komunikasi dengan database Supabase, seperti mengambil data (`getAll()`), mencari data berdasarkan ID (`getById()`), menambahkan data (`add()`), memperbarui data (`updateItem()`), dan menghapus data (`deleteItem()`).

Dengan pendekatan ini, apabila di kemudian hari sumber data berubah, misalnya dari Supabase menjadi REST API atau SQLite, perubahan cukup dilakukan pada kelas repository tanpa memengaruhi bagian lain dari aplikasi.

---

## Keuntungan Penggunaan Repository Pattern

1. Memisahkan logika akses data dari logika aplikasi.
2. Membuat kode lebih terstruktur dan mudah dipelihara.
3. Mempermudah pergantian sumber data tanpa mengubah kode pada lapisan lain.
4. Mendukung prinsip **Separation of Concerns** sehingga setiap kelas memiliki tanggung jawab yang jelas.
5. Memudahkan proses pengujian (testing) karena repository dapat diganti dengan implementasi lain atau mock object.

## Kesimpulan

Berdasarkan hasil analisis, proyek tugas besar ini menerapkan **Repository Pattern** melalui penggunaan `BaseRepository<T>` sebagai abstraksi dan `ProjectRepository` sebagai implementasi akses data. Pattern ini membuat pengelolaan data menjadi lebih modular, mudah dikembangkan, dan lebih fleksibel terhadap perubahan sumber data di masa mendatang.
